# Subscriptions And Billing

Subscription and recurring billing are a separate product domain. Billing creates invoices and off-session payment obligations; routing decides how eligible payment attempts execute.

## Ownership

MasonXPay owns subscription plans, subscription lifecycle, invoices, invoice payment attempts, retry/dunning state, and merchant operations. Provider billing products can be added later as explicit connector-specific adapters, but they should not be the default core model.

Provider responsibilities:

- run customer-present setup or first-payment flows
- return safe reusable payment references
- execute off-session payment attempts
- emit webhooks for reconciliation

MasonXPay responsibilities:

- customer records and customer payment-method references
- subscription periods, items, cancellation, and lifecycle rules
- invoice generation and invoice lifecycle
- recurring retry policy and dunning state
- deterministic routing of invoice payment attempts

## Payment Method Boundary

Subscription checkout converts customer-present provider tokens or nonces into provider-scoped `PaymentInstrument` rows that contain only safe reusable provider references. Later invoices charge those saved references off-session through normal payment services.

Provider-scoped reusable instruments can only be charged through their original provider account unless a future portable vault or network-token integration explicitly supports broader routing.

## Module Shape

Keep billing under focused boundaries:

- `billing/customer`
- `billing/paymentmethod`
- `billing/subscription`
- `billing/invoice`
- `billing/worker`
- `billing/dunning`
- `billing/web`

Billing should not shortcut into provider adapters. It should call payment services using normal payment intent and instrument abstractions.

## Retry Boundary

Recurring retry is not customer-present checkout retry and not refund retry. It applies only to invoices generated by an active subscription schedule. Automatic retries must be bounded by merchant policy, attempt count, schedule, and terminal actions.

Refund auto-retry remains disabled by default unless an explicit approval workflow is introduced.
